## ℹ️ О проекте

Данный проект представляет собой набор инструментов для автоматизации процесса синхронизации между двумя Git репозиториями. Разработанные скрипты решают типичную задачу разработчика: обновление исходного репозитория, перенос файлов в целевой репозиторий и автоматическая отправка изменений на удаленный сервер. Решение представлено в двух вариантах, что позволяет выбрать наиболее подходящий инструмент в зависимости от среды выполнения и требований к функциональности.

### 🎯 Цели проекта

| Цель | Описание |
|------|----------|
| **Автоматизация** | Минимизация ручного вмешательства в процесс синхронизации |
| **Надежность** | Гарантия целостности данных при переносе между репозиториями |
| **Гибкость** | Поддержка различных сценариев использования |
| **Прозрачность** | Детальное логирование всех выполняемых операций |

### 👥 Целевая аудитория

- **Разработчики**, работающие с несколькими репозиториями одновременно
- **DevOps инженеры**, настраивающие CI/CD процессы
- **Студенты**, изучающие системы контроля версий
- **Команды**, нуждающиеся в регулярной синхронизации кода

## 🔄 Общая концепция работы

Оба скрипта реализуют единый алгоритм, состоящий из трех последовательных этапов, каждый из которых выполняет критически важную функцию в процессе синхронизации.

### 📊 Схема процесса

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   Этап 1        │────▶│   Этап 2        │────▶│   Этап 3        │
│   Git Pull      │     │   Копирование   │     │   Git Push      │
│   (Обновление)  │     │   (Перенос)     │     │   (Отправка)    │
└─────────────────┘     └─────────────────┘     └─────────────────┘
```

### ⚙️ Поток данных

1. **Входные данные**: Пути к двум репозиториям, сообщение коммита
2. **Обработка**: Обновление, копирование, фильтрация, коммит
3. **Выходные данные**: Синхронизированный целевой репозиторий на сервере

## 📥 Этапы синхронизации

### Первый этап: Обновление исходного репозитория

На этом этапе скрипт переходит в директорию с исходным репозиторием и выполняет команду Git pull. Это действие гарантирует, что все файлы, которые будут скопированы, находятся в актуальном состоянии и содержат последние изменения, полученные из удаленного репозитория.

| Действие | Команда | Назначение |
|----------|---------|------------|
| Переход в директорию | `cd $SOURCE_REPO` | Смена рабочей папки |
| Проверка пути | `test -d "$SOURCE_REPO"` | Валидация существования |
| Получение изменений | `git pull` | Синхронизация с сервером |

### Второй этап: Копирование файлов между репозиториями

После обновления исходного репозитория скрипт приступает к копированию всех файлов и папок из исходной директории в целевую. Важной особенностью этого этапа является исключение служебной папки Git, что предотвращает повреждение метаданных целевого репозитория.

### 📁 Исключаемые файлы и папки

```bash
.git/                    # Служебная папка Git
.gitignore               # Файл игнорирования
*.log                    # Файлы логов
*.tmp                    # Временные файлы
.DS_Store                # macOS системный файл
Thumbs.db                # Windows кэш миниатюр
```

### Третий этап: Отправка изменений в удаленный репозиторий

Завершающий этап выполняется в директории целевого репозитория. Скрипт последовательно выполняет команду Git add для добавления всех изменений в индекс, Git commit для создания коммита с заданным сообщением и Git push для отправки изменений в удаленный репозиторий.

### 🔄 Последовательность команд

```bash
git add .                     # Добавление всех изменений в индекс
git commit -m "$MESSAGE"      # Создание коммита с сообщением
git push origin "$BRANCH"     # Отправка в удаленный репозиторий
```

## 🐚 Скрипт для Git Bash

### Назначение и область применения

Скрипт для Git Bash представляет собой легковесное решение, ориентированное на пользователей, которые предпочитают Unix подобную командную строку и ценят простоту. Он идеально подходит для быстрой ручной синхронизации, когда не требуется сложная обработка ошибок или дополнительные проверки.

### 📁 Структура и компоненты

```bash
#!/bin/bash
# ==============================================================================
# СКРИПТ АВТОМАТИЗАЦИИ GIT (ДЛЯ GIT BASH НА WINDOWS)
# ==============================================================================
# Версия: 1.0
# ==============================================================================

# ------------------- 🔧 НАСТРОЙКИ -------------------------

SOURCE_REPO="C:\Users\111\mfua"
TARGET_REPO="/c/Users/111/mfua-learning"
COMMIT_MESSAGE="update"
TARGET_BRANCH="README"

# ----------------------------------------------------------
```

### 📝 Описание переменных

| Переменная | Тип | Обязательная | Описание |
|------------|-----|--------------|----------|
| `SOURCE_REPO` | string | ✅ | Путь к исходному репозиторию |
| `TARGET_REPO` | string | ✅ | Путь к целевому репозиторию |
| `COMMIT_MESSAGE` | string | ✅ | Текст сообщения коммита |
| `TARGET_BRANCH` | string | ✅ | Имя ветки для отправки изменений |

### 🎨 Визуальное оформление вывода

```bash
GREEN='\033[0;32m'
YELLOW='\033[1;33m'
CYAN='\033[0;36m'
NC='\033[0m'

echo -e "${CYAN}========================================${NC}"
echo -e "${CYAN}🚀 Запуск синхронизации (Git Bash)${NC}"
echo -e "${CYAN}========================================${NC}"
```

### 📊 Цветовая схема

| Цвет | Код | Назначение |
|------|-----|------------|
| 🟢 Зеленый | `\033[0;32m` | Успешные операции |
| 🟡 Желтый | `\033[1;33m` | Текущий выполняемый шаг |
| 🔵 Синий | `\033[0;36m` | Заголовки и разделители |
| 🔴 Красный | `\033[0;31m` | Ошибки |

### 🔄 Реализация этапа обновления

```bash
echo -e "\n${YELLOW}📥 Шаг 1: Git pull...${NC}"
cd "$SOURCE_REPO" || { echo "❌ Ошибка перехода"; exit 1; }
git pull
echo -e "${GREEN}✅ Этап 1 завершен${NC}"
```

### 📁 Реализация этапа копирования

```bash
echo -e "\n${YELLOW}📋 Шаг 2: Копирование файлов...${NC}"
cp -rf ./* "$TARGET_REPO/" 2>/dev/null
echo -e "${GREEN}✅ Файлы скопированы${NC}"
```

### 🚀 Реализация этапа отправки

```bash
echo -e "\n${YELLOW}📤 Шаг 3: Git add, commit, push...${NC}"
cd "$TARGET_REPO" || { echo "❌ Ошибка перехода"; exit 1; }

git add .
git commit -m "$COMMIT_MESSAGE"
git push origin "$TARGET_BRANCH"

echo -e "\n${GREEN}🎉 Готово!${NC}"
```

## 💻 Скрипт для PowerShell

### Назначение и архитектура

PowerShell скрипт представляет собой полнофункциональное решение с модульной архитектурой. В отличие от Bash версии, он включает множество дополнительных возможностей: расширенную обработку ошибок, режим имитации, автоматическое разрешение конфликтов, сохранение конфигурации и детальное логирование.

### 📋 Параметры командной строки

```powershell
param(
    [string]$Repo1Path = "",
    [string]$Repo2Path = "",
    [string]$CommitMessage = "Sync $(Get-Date -Format 'yyyy-MM-dd HH:mm:ss')",
    [switch]$CleanMode,
    [switch]$DryRun,
    [switch]$ForcePush
)
```

### 📊 Описание параметров

| Параметр | Тип | Значение по умолчанию | Описание |
|----------|-----|----------------------|----------|
| `Repo1Path` | string | "" | Путь к исходному репозиторию |
| `Repo2Path` | string | "" | Путь к целевому репозиторию |
| `CommitMessage` | string | Авто с датой | Сообщение для коммита |
| `CleanMode` | switch | $false | Очистка целевой папки |
| `DryRun` | switch | $false | Режим имитации |
| `ForcePush` | switch | $false | Принудительный пуш |

### 📝 Функция логирования

```powershell
function Write-Log {
    param(
        [string]$Message,
        [string]$Color = "White"
    )
    $timestamp = Get-Date -Format "yyyy-MM-dd HH:mm:ss"
    Write-Host "[$timestamp] $Message" -ForegroundColor $Color
}
```

### 🔧 Функция выполнения Git команд

```powershell
function Invoke-GitCommand {
    param(
        [string]$Command,
        [string[]]$Arguments,
        [string]$WorkingDirectory,
        [switch]$ShowOutput = $true
    )
    
    try {
        Push-Location $WorkingDirectory
        $cmdString = "git $Command " + ($Arguments -join " ")
        
        if ($DryRun) {
            Write-Log "[DRY RUN] $cmdString" "Yellow"
            return $true
        }
        
        $output = & git $Command @Arguments 2>&1
        $exitCode = $LASTEXITCODE
        
        if ($exitCode -ne 0) {
            Write-Log "Error: $cmdString" "Red"
            return $false
        }
        
        return $true
    } catch {
        Write-Log "Exception: $_" "Red"
        return $false
    } finally {
        Pop-Location
    }
}
```

### 📁 Функция копирования файлов

```powershell
function Copy-FilesBetweenRepos {
    param(
        [string]$SourcePath,
        [string]$DestPath,
        [switch]$Clean
    )
    
    $excludeItems = @(".git", ".gitignore", "*.log", "*.tmp", ".DS_Store")
    
    if ($Clean -and -not $DryRun) {
        Get-ChildItem -Path $DestPath -Force | 
        Where-Object { $_.Name -ne ".git" } | 
        ForEach-Object { Remove-Item -Path $_.FullName -Recurse -Force }
    }
    
    Get-ChildItem -Path $SourcePath -Force | 
    Where-Object { $_.Name -ne ".git" } | 
    ForEach-Object {
        Copy-Item -Path $_.FullName -Destination $DestPath -Recurse -Force
    }
    
    Write-Log "Copy completed" "Green"
    return $true
}
```

### 🔄 Основная функция синхронизации

```powershell
function Sync-Repositories {
    Write-Log "Starting repository synchronization" "Magenta"
    
    # Проверка путей
    if (-not (Test-Path $Repo1Path)) {
        Write-Log "Error: Repository 1 not found" "Red"
        return $false
    }
    
    # Этап 1: Обновление источника
    Invoke-GitCommand -Command "pull" -WorkingDirectory $Repo1Path
    
    # Этап 2: Обновление назначения
    Invoke-GitCommand -Command "pull" -WorkingDirectory $Repo2Path
    
    # Этап 3: Копирование файлов
    Copy-FilesBetweenRepos -SourcePath $Repo1Path -DestPath $Repo2Path
    
    # Этап 4: Отправка изменений
    Invoke-GitCommand -Command "add" -Arguments @(".") -WorkingDirectory $Repo2Path
    Invoke-GitCommand -Command "commit" -Arguments @("-m", $CommitMessage) -WorkingDirectory $Repo2Path
    Invoke-GitCommand -Command "push" -WorkingDirectory $Repo2Path
    
    Write-Log "Synchronization completed" "Green"
}
```

### 💾 Сохранение и загрузка конфигурации

```powershell
function Save-Config {
    $config = @{
        Repo1Path = $Repo1Path
        Repo2Path = $Repo2Path
        LastSync = Get-Date -Format "yyyy-MM-dd HH:mm:ss"
    }
    $config | ConvertTo-Json | Set-Content -Path "reposync.config.json"
}

function Load-Config {
    if (Test-Path "reposync.config.json") {
        $config = Get-Content "reposync.config.json" | ConvertFrom-Json
        $script:Repo1Path = $config.Repo1Path
        $script:Repo2Path = $config.Repo2Path
    }
}
```

## 🔧 Устранение неполадок

### ❌ Ошибка: "Permission denied"

| Причина | Решение |
|---------|---------|
| Нет прав на запись | Запустить от имени администратора |
| Файл заблокирован | Закрыть программу, использующую файл |

### ❌ Ошибка: "git: command not found"

| Причина | Решение |
|---------|---------|
| Git не установлен | Установить Git с git-scm.com |
| Git не в PATH | Добавить Git в переменную PATH |

### ❌ Ошибка: "Authentication failed"

| Причина | Решение |
|---------|---------|
| Неверный пароль | Использовать Personal Access Token |
| Истек токен | Создать новый токен в GitHub Settings |

### ❌ Ошибка: "Conflict during merge"

| Причина | Решение |
|---------|---------|
| Параллельные изменения | Выполнить manual merge |
| Удаленные изменения | Сделать git pull перед push |

## ❓ Часто задаваемые вопросы

**Q: Можно ли использовать скрипт с приватными репозиториями?**  
**A:** Да, скрипт работает с любыми репозиториями, к которым у вас есть доступ.

**Q: Как настроить автоматический запуск по расписанию?**  
**A:** Windows: Task Scheduler, Linux/macOS: cron

**Q: Что делать если скрипт зависает на git pull?**  
**A:** Проверьте подключение к интернету и доступность Git сервера.

**Q: Как откатить изменения если что-то пошло не так?**  
**A:** Используйте `git reset --hard HEAD~1` для отката последнего коммита.
